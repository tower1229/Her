# Timeline v2 可落地性复盘（面向完整实现）

> 状态：实现性复盘 / 设计纠偏备忘
> 目的：从“这个系统最终能不能完整落地”出发，重新评估 Timeline v2 还缺什么、哪些地方必须改、哪些设计已经具备工程可行性
> 读者：项目维护者、未来的 persona-skill 设计者、以及要把 Timeline 真正接进 OpenClaw 的实现者

## 1. 先说结论

如果 Timeline 的目标只是：
- 让 OpenClaw 回答“我现在在干嘛”；
- 给一个结构化 JSON；
- 顺便往 `memory/` 里写一条日志；

那么现在的系统已经差不多够了。

但如果 Timeline 的真正目标是：

> **让 OpenClaw 拥有像真人一样、能与 persona system 深度耦合的自传式回忆能力，并且让闲聊、自拍、未来的 persona-skill、乃至更复杂的消费端都能使用同一条记忆现实。**

那么当前系统虽然已经进入“可继续构建”的阶段，但还没有到“完整可落地”的程度。

关键原因不是代码量不够，而是**系统边界和生成架构还需要再明确一层**。

---

## 2. 这件事真正要落地，必须满足什么条件

Timeline 真正落地，不是把一个工具 call 写出来就结束了，而是要同时满足下面四件事：

### 2.1 能读到真实事实锚点

也就是：
- `sessions_history` 能提供当前轮次与近邻轮次的硬事实；
- `memory_get` 能读到已落盘的日记 canon；
- `memory_search` 能提供长期语义补充；
- `SOUL` / `MEMORY` / `IDENTITY` 能作为人格化生成的高价值语义输入。

如果这些输入拿不到，Timeline 就不可能真正生成“像这个角色会经历过的记忆”。

### 2.2 能在空白窗口下做真正的人格化生成

这一步**不适合完全靠脚本硬编码**。
原因非常直接：
- `SOUL` 和 `MEMORY` 会越来越定制化；
- 未来 persona-skill 会持续改写这些内容；
- 这些文本的语义强度远高于脚本关键词匹配能可靠处理的程度。

所以从完整实现角度看，Timeline 必须有一个**LLM 驱动的生成插槽（generation slot）**：
- 它把 `SOUL` / `MEMORY` / `IDENTITY` / timeline facts / 当前真实时间等内容整理给模型；
- 模型负责做高语义密度的人格化推断；
- Timeline runtime 负责验证、归一化、写盘、留 trace。

### 2.3 能把生成结果稳定写成 canon

如果生成之后只是拿来回复一句话，而不写入 canon，那么：
- 闲聊会漂；
- 自拍不会和之前的状态对齐；
- persona 也无法依赖长期一致的生活轨迹。

所以 Timeline 必须继续维持“**统一写盘责任方**”这个原则。

### 2.4 消费端真的能稳定消费

最终消费 Timeline 的并不是只有聊天回复。
未来至少还包括：
- 自拍类 skill；
- 未来的 persona-skill；
- 其他需要“当前状态 / 最近状态 / 某天状态”的场景化技能。

这意味着 Timeline 输出结构不能只从“内部工程调试方便”出发，还必须从“消费端稳定拿字段”出发设计。

---

## 3. 当前这套系统已经具备哪些可落地基础

### 3.1 统一入口已经成立

这是一个非常重要的正面结论。
`timeline_resolve` 已经把：
- 读事实；
- 判定是否命中既有 canon；
- 需要时生成；
- 写入；
- trace / status 记录；

收敛到一条 runtime 主路径里。

这意味着 Timeline 已经不再是 prompt 文案拼装，而是有了真正的系统骨架。

### 3.2 统一写盘责任已经成立

Append-only 写盘、路径校验、锁保护这些东西虽然还没到最终形态，但方向已经正确。
这为后面接入更强的 LLM 生成创造了前提：
**模型可以负责“想”，runtime 负责“写”和“约束”。**

### 3.3 结构化消费载荷仍然可用

虽然当前工具 envelope 比早期 prompt 时代更强，但 `timeline.window` + `Episode` 这套核心消费载荷仍然存在。
这说明和 persona / selfie 的接口并没有被彻底破坏，仍然有可延续性。

---

## 4. 目前仍然阻碍“完整落地”的关键缺口

### 4.1 缺少真正接入 OpenClaw LLM 的生成实现

当前代码已经加上了一个**LLM generation seam**，这是正确方向。
但从可落地角度说，真正还没完成的是：
- 这条 seam 如何接 OpenClaw 的模型调用接口；
- prompt / tool / response schema 如何在真实 runtime 中稳定执行；
- 失败时如何回退；
- 生成结果如何做结构验证与安全约束。

也就是说：**“能接 LLM”这件事的架构已经对了，但“真正接上并验证”还没完成。**

### 4.2 persona-skill 与 Timeline 的协作边界还需要正式定义

未来 persona-skill 会改写 `SOUL` 和 `MEMORY`，这意味着：
- 哪些字段由 persona 决定；
- 哪些事实由 Timeline 决定；
- Timeline 什么时候应当相信 persona 文本；
- persona 又该如何消费 Timeline 已经落盘的 canon；

这些边界必须写成明确契约，否则后面很容易出现“人格设定说 A，timeline canon 记录 B，自拍又渲染成 C”的分裂现象。

### 4.3 多日窗口还没有形成最终完成态

原始设计里，`recent_3d` / `explicit` 更理想的形态是“按日拆分后逐日处理”。
现在虽然 blank-memory generation 已不再被粗暴禁用，但多日窗口的最终返回形态和写盘策略还没有真正走到最后一版。

这部分如果不收口，未来消费端在拿“最近三天状态”时仍会存在歧义。

### 4.4 输出结构还需要一次“消费视角”审视

现在的 envelope 里有：
- `ok`
- `trace_id`
- `resolution_summary`
- `notes`
- `trace`

这些字段对 runtime 与调试是有价值的。
但从最终系统角度，必须明确区分：
- 哪些是**消费端真正依赖的稳定协议**；
- 哪些只是 runtime 内部可选元信息。

否则未来 persona / selfie / 其他技能会被迫依赖太多本不该依赖的字段。

---

## 5. 这轮设计纠偏之后，方向哪里变正确了

### 5.1 生成已经不再被错误地理解为“脚本自己想象一下”

本轮最大的纠偏点，是承认：

> **真正的人格化记忆生成，核心应该由 LLM 完成；runtime 负责约束、持久化、追踪，而不是试图用脚本完全替代语义理解。**

这是从“能解释”走向“能落地”的关键一步。

### 5.2 runtime 角色被重新摆正

Timeline runtime 最合理的角色应该是：
- 组织上下文；
- 保证读取顺序；
- 决定是否需要生成；
- 调用生成器；
- 验证结构；
- 统一写盘；
- 保留 trace。

而不是：
- 自己硬编码所有人格逻辑；
- 自己试图理解任意复杂的 `SOUL` 文本。

### 5.3 对未来 persona-skill 留出了正确扩展位

一旦 Timeline 的生成插槽明确是 LLM 驱动，那么未来 persona-skill 就可以放心把更多人格化、定制化内容放进 `SOUL` / `MEMORY`：
- Timeline 不需要知道这些内容的内部模板；
- 只需要把它们作为语义上下文交给 LLM；
- 再把输出固定成 Timeline 的结构化结果即可。

这才是能长期演化的系统结构。

---

## 6. 如果目标是“真的做成”，接下来最应该补的不是哪里

最不应该继续花大量时间的方向，是：
- 继续往脚本 heuristics 里堆更多关键词；
- 试图把所有人格逻辑都硬编码进 `infer_candidate.ts`；
- 继续只围绕“文档能不能解释清楚”打转。

这些工作只能让系统“看起来更完整”，但不能让它真正成为未来 persona system 的稳定基础。

---

## 7. 如果目标是“真的做成”，接下来最应该做什么

### 第一优先级：把 LLM generation seam 接成真实能力

要完成：
1. 确认 OpenClaw 中可用于该插件的模型调用面；
2. 把 Timeline generation prompt / schema 变成真实可调用接口；
3. 为模型输出增加结构验证与失败回退；
4. 对生成结果做写盘前的安全检查。

### 第二优先级：补 persona-skill / Timeline 边界契约

要完成：
1. 规定 persona 可以影响什么；
2. 规定 timeline canon 的事实优先级；
3. 规定自拍类技能应该依赖哪些字段；
4. 写清楚 persona 改设定后 timeline 如何承接。

### 第三优先级：完成多日窗口的最终设计

要完成：
1. `recent_3d` / `explicit` 是否返回 `days[]`；
2. 多日生成与多日只读命中的混合情况如何表达；
3. 消费端如何稳定消费这类返回。

### 第四优先级：把消费协议和诊断协议分层

要完成：
1. 定义稳定消费字段；
2. 定义调试/trace字段；
3. 避免下游误依赖 runtime 元信息。

---

## 8. 最终判断：这套系统能不能被完整实现？

答案是：**能，但前提是必须承认 Timeline 是一个“LLM + runtime 约束”的复合系统，而不是单纯脚本系统。**

如果沿着下面这条路走，这个系统是可落地的：
- timeline runtime 负责系统秩序；
- LLM 负责语义级人格化生成；
- persona-skill 负责持续塑造 `SOUL` / `MEMORY`；
- Timeline 负责把这些人格化语义与真实时间线、既有 canon、写盘规则统一起来；
- 下游消费端统一吃 `timeline.window` / `Episode` 的稳定结果。

如果反过来，坚持让脚本自己“理解”越来越复杂的 persona 设定，那么这个系统最终只会停留在 demo 阶段，而很难成为一个可持续演化的 OpenClaw 人格记忆层。

---

## 9. 一句话总结

> **Timeline 真正能落地的关键，不是把更多 heuristics 塞进脚本，而是把它升级成“LLM 语义生成 + runtime 确定性约束 + 统一写盘 + 稳定消费协议”的完整系统。**
