# Timeline 路线图

> 状态：面向北极星目标的执行路线图
> 目的：把 Timeline 从“可调用 runtime”推进到“OpenClaw 的时间现实层”
> 读者：项目维护者与实现者
> 关联：`docs/timeline-north-star.md`

## 1. 路线图原则

这份路线图不以“再补几个工具细节”为目标，而以北极星目标为目标：

> 让 OpenClaw 在自然聊天和下游技能中，都共享一条连续、可信、可写盘、可追踪的时间现实。

因此路线图的优先级应是：

1. 先定义体验与契约
2. 再重做 resolver 语义
3. 再接入真实 LLM 生成
4. 再打通下游 skill 联动
5. 最后再围绕发布与运维做硬化

同时增加一条执行约束：

- 不允许把本应由 LLM 完成的语义工作，继续包装成脚本 fallback 保留在项目里。
- 不以“改动最小”为默认目标，而以“是否符合北极星目标”为唯一优先级。

## 2. 阶段划分

## 阶段 P0：冻结北极星目标与验收口径

### 目标

把“工具能跑通”彻底升级成“体验是否成立”的评估方式。

### 交付物

- 北极星设计文档
- LLM / 脚本职责边界文档
- collector / reasoner 接口草案
- 路线图文档
- 统一的体验型验收清单
- 文档语言与设计约束

### 关键任务

- 明确聊天、记忆、连续性、自拍联动四类主场景
- 把这些场景转成 L3/L4 级验收标准
- 停止把显式 tool call 当成最终成功标准
- 固化 LLM 与脚本的职责边界

### 完成标准

- 项目内有明确的最高设计目标
- 后续任务都能映射回该目标
- 评估口径从 tool-centric 切换到 experience-centric

## 阶段 P1：重做 Timeline resolver 语义

### 目标

让 `timeline_resolve` 从“查当天最后一条或空则生成”升级成“按问题找事实、必要时补洞、保持连续性”的真正时间决策引擎。

### 当前主要问题

- 命中策略仍偏向当天最后一条
- `recent_3d` / `explicit` 的最终语义还未收口
- 已有日内事实但当前窗口缺事实时，不会做 gap-fill generation
- 连续性推理没有独立建模
- 当前部分读取决策仍然试图用脚本承担语义判断

### 关键任务

- 把窗口到候选 facts 的收集，与最终语义决策彻底拆开
- 让 episode 选择与连续性判断回到 LLM 决策层
- 引入“目标窗口命中 / 窗口缺口 / 延续推理 / 新生成”四类决策
- 把 `recent_3d` 和跨日 explicit 窗口正式建成多日结构
- 为事件持续性建立最小状态模型

### 建议实现切面

- `src/core/resolve_window.ts`
- `src/core/map_window.ts`
- `src/tools/timeline_resolve.ts`
- 新增 collector / reasoner 分层相关模块

### 完成标准

- “你在干嘛” 不再只是复用当天最后一条
- “最近在做什么” 能稳定按窗口组织结果
- “五分钟前打球，五分钟后还在打球” 不再依赖脚本关键词或活动时长表

## 阶段 P2：把 LLM generation seam 接成真实能力

### 目标

让 Timeline 具备真正的人格化生成能力，而不是主要依赖 heuristics 占位。

### 当前主要问题

- `generateMemoryDraft` 只是 seam，不是完整产品能力
- heuristics 仍承担了过多语义职责
- 模型输出验证、失败回退和写盘前安全检查还不够完整

### 关键任务

- 确认 OpenClaw 中可供插件使用的模型调用接口
- 固化 generation prompt 和 response schema
- 增加模型输出结构校验与归一化
- 增加生成失败回退与置信度策略
- 明确哪些情况下允许写盘，哪些情况下只返回不写
- 删除脚本式人格化生成与相关 fallback 代码，而不是继续保留为“兜底”

### 建议实现切面

- `src/core/generation_prompt.ts`
- `src/tools/timeline_resolve.ts`
- 新增 model-backed generation adapter
- 不再保留旧的脚本式人格化生成模块，生成功能只保留 LLM draft 的结构化落盘路径

### 完成标准

- 空白窗口能生成更像真人的状态与回忆
- 生成结果和 persona 文本有明显语义耦合
- 项目中不再保留脚本式人格化生成 fallback

## 阶段 P3：定义稳定消费协议与下游联动

### 目标

让 Timeline 成为聊天层和下游 skill 的统一时间现实来源。

### 当前主要问题

- 输出 envelope 偏 runtime 调试视角
- 自拍 / persona 等消费端依赖字段尚未正式定义
- 缺少“聊天自然语言回复”和“下游结构化消费”的分层协议

### 关键任务

- 区分消费协议与诊断协议
- 定义下游 skill 稳定字段
- 定义聊天层如何把 Timeline 结果自然语言化
- 做第一条 timeline -> selfie 的真实联调链路

### 建议消费字段

- `window`
- `episodes`
- 地点、行为、情绪、外观、可信度
- 事实来源类型
- 当前状态是否为延续推理 / 新生成 / 只读命中

### 完成标准

- 自拍 skill 可以稳定使用 Timeline 的当前状态
- persona skill 不再各自生成一套冲突状态
- 用户在聊天与下游产物里看到的是同一现实

## 阶段 P4：建立体验型 E2E 测试体系

### 目标

让测试真正覆盖北极星目标，而不是只覆盖局部工具行为。

### 当前主要问题

- 当前 smoke 更偏注册、加载、显式工具调用
- 缺少自然问法驱动的 agent E2E
- 缺少 timeline 与下游技能联调回归

### 关键任务

- 把 L3 测试主场景改成自然语言问法
- 增加连续性追问测试
- 增加自拍联动测试
- 增加“事实优先 / 生成补空 / canon 延续”的端到端断言

### 建议验收场景

- “你在干嘛”
- “最近有什么有趣的事吗”
- “你刚才不是在打球吗，现在还在吗”
- “给我来一张你现在状态的自拍”

### 完成标准

- 不要求用户显式说“请调用 timeline_resolve”
- 真正由 agent 自行路由并产出自然体验
- 失败时 trace 能解释清楚是路由失败、检索失败还是生成失败

## 阶段 P5：发布与运维硬化

### 目标

在体验闭环基本成立后，再把 Timeline 推向更稳定的发布状态。

### 关键任务

- 继续强化单写者与并发保护
- 增强 repair 工作流
- 完善状态页与 trace 可读性
- 明确版本升级策略与兼容约束

### 完成标准

- 不是只“能跑”，而是“可持续维护”
- 出错后维护者能低成本定位问题
- 发布标准围绕体验与可维护性双重通过

## 3. 近期三轮建议排期

### 第一轮

- 完成 P0
- 启动 P1 的 resolver 语义重构设计
- 明确连续性推理的最小模型

### 第二轮

- 完成 P1 核心实现
- 启动 P2 的真实 LLM generation 接入
- 明确下游消费字段草案

### 第三轮

- 打通 P2 与 P3
- 做 timeline -> selfie 第一条真实链路
- 建立第一组体验型 E2E 用例

## 4. 本阶段不应继续投入过多精力的方向

- 继续往 heuristics 里堆大量关键词
- 以“先留一个脚本 fallback 以防万一”为理由保留旧智能层代码
- 围绕显式 tool call 成功反复做展示式测试
- 在消费协议未定前，让下游 skill 直接依赖当前 trace/envelope 细节
- 过早追求“文档看起来齐全”，却不收敛北极星目标

## 5. 项目推进口令

以后每一个任务都应该先回答这三个问题：

1. 它是否增强了 OpenClaw 的真人时间感知与回忆体验？
2. 它是否让聊天与下游 skill 更共享同一条时间现实？
3. 它是否在事实、连续性、人格化生成之间建立了更清晰的系统秩序？

如果三个问题都答不上来，这个任务优先级就应该下降。

补充一条：

4. 它是不是在用脚本替代本应由 LLM 完成的语义理解？

如果答案是“是”，那它通常不是正确方案，即使看起来改动更小。
