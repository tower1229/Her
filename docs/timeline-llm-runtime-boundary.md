# Timeline 中 LLM 与脚本的职责边界

> 状态：当前有效
> 目的：明确 Timeline 在最高设计目标下，哪些职责必须由 LLM 实现，哪些职责必须由脚本实现，避免继续在错误边界上做局部修补
> 关联：`docs/timeline-north-star.md`、`docs/timeline-roadmap.md`

## 1. 这份文档要解决什么问题

Timeline 当前面临的最大风险，不是“代码还不够多”，而是：

> 把本应由 LLM 完成的高语义工作，错误地压给脚本去做。

一旦边界划错，就会不断出现以下现象：

- 用关键词匹配理解用户时间问题
- 用活动类型表推理状态持续多久
- 用脚本规则在多个 canon episode 里选“最相关”那一条
- 用 heuristics 代替 persona 理解和回忆组织

这些方案都可能暂时跑通，但它们不会收敛到项目真正的目标。

## 2. 总原则

Timeline 的分工必须遵守一句话：

> LLM 负责理解、判断与组织；脚本负责约束、验证与执行。

如果一段逻辑的正确性依赖：
- 自然语言理解
- 常识推理
- persona 语义理解
- 多事实之间的语义判断
- 回忆的组织与取舍

那么它就不应该由脚本主导。

## 3. 必须由 LLM 实现的职责

### 3.1 时间意图理解

LLM 必须判断用户当前是在问：
- 当前状态
- 近期回忆
- 明确过去时间
- 连续性追问
- 下游 skill 所需的当前状态提取

原因：
- 语言表达高度多样
- 不存在可靠的可枚举关键词表

### 3.2 查询语义到事实选择

当已有多个 canon facts 时，LLM 必须判断：
- 哪一条最相关
- 是否应组合多条
- 是否虽有历史事实，但对当前问题仍然不构成命中

原因：
- “最相关”本质上是语义问题，不是 token 命中问题

### 3.3 连续性与状态转移推理

LLM 必须判断：
- 先前活动是否仍在持续
- 当前时间是否已超过合理持续窗口
- 是否应从“打球”转入“休息”或“回家”

原因：
- 这是常识推理，不是稳定规则表

### 3.4 空白窗口的人格化生成

LLM 必须基于：
- `SOUL`
- `MEMORY`
- `IDENTITY`
- 既有 timeline canon
- 当前真实时间
- 周末 / 节日 / 日内时段

生成符合 persona 且符合现实的状态或回忆。

### 3.5 近期回忆的组织与摘要

像“最近有什么有趣的事吗”这种问题，LLM 必须负责：
- 取舍重点
- 组织叙事
- 保持像回忆而不是像日志枚举

## 4. 必须由脚本实现的职责

### 4.1 数据收集

脚本必须负责：
- 读取 `sessions_history`
- 读取 `memory_get`
- 读取 `memory_search`
- 读取 `SOUL` / `MEMORY` / `IDENTITY`
- 整理候选 facts 与时间锚点

### 4.2 硬边界与基础归一化

脚本必须负责：
- `explicit` 输入是否合法
- 时区与时间格式归一化
- 当前 anchor 时间注入
- canonical 路径校验

这些是规则执行，不是语义理解。

### 4.3 硬事实优先级约束

脚本必须强制：
- 会话硬事实优先
- 已落盘 canon 不可被随意改写
- memory_search 不能否定硬事实
- persona 文本不能覆盖已成立的时间事实

### 4.4 输出结构校验

脚本必须验证 LLM 输出是否满足 Timeline 结构要求，例如：
- `location`
- `action`
- `emotionTags`
- `appearance`
- `confidence`

以及这些字段是否可安全写盘。

### 4.5 写盘与幂等

脚本必须负责：
- append-only 写盘
- 锁
- 并发冲突
- 指纹 / 幂等
- trace
- status
- repair

### 4.6 失败处理

当 LLM 不可用、返回不合法、或与硬事实冲突时：
- 脚本可以拒绝写盘
- 脚本可以拒绝将其认定为 canon
- 脚本可以返回明确失败

但脚本不能自己接管生成与判断。

## 5. 不允许脚本主导的职责

以下做法从现在开始应视为错误方向：

- 用关键词表判断“你在干嘛”属于哪种时间问题
- 用脚本 token 匹配在多个 episode 中选最相关事实
- 用活动时长表推理“打球还能持续多久”
- 用 heuristics 生成 persona 状态
- 当 LLM 缺席时让脚本产出“差不多合理”的内容

这些都属于临时过渡件，不应成为正式设计的一部分。

## 6. Timeline 的推荐分层

按照当前目标，Timeline 应收敛成四层：

### Layer 1：Collector

脚本层，负责：
- 收集事实源
- 生成候选 episode 集
- 整理 persona 上下文
- 构建时间锚点

### Layer 2：Temporal Reasoner

LLM 层，负责：
- 识别请求类型
- 判断是否命中既有事实
- 判断是否属于持续性追问
- 决定是否需要生成
- 给出结构化决策结果

### Layer 3：Runtime Guard

脚本层，负责：
- 验证 LLM 决策是否违反硬事实
- 验证结构是否完整
- 判断是否允许写盘

### Layer 4：Canon Writer

脚本层，负责：
- append-only 写盘
- 冲突处理
- 运行状态持久化
- trace 与 repair

## 7. 对当前代码的直接影响

按这份边界文档，以下类型的实现应被视为临时过渡件，而不是最终方案：

- 用脚本直接做 query 关键词匹配来选 facts
- 用脚本枚举活动类别并推理持续时长
- 用脚本判断用户问题属于 current_status 还是 recall

它们可以作为短期辅助调试存在于过渡阶段，但不应继续扩展，更不应被当成正式设计收口。

## 8. 重构原则

后续重构必须遵守下面三条：

1. 优先采用与设计目标一致的正确方案，而不是改动最小的方案。
2. 如果一个模块的职责边界本身错了，就应直接重构或删除，而不是继续在原地打补丁。
3. 任何为“先跑通再说”而引入的脚本语义层，都必须被视为待删除，而不是待长期保留。

## 9. 一句话总纲

> Timeline 不是“脚本替 LLM 做语义推理，再让 LLM 润色”的系统，而是“LLM 做语义推理，脚本做约束与执行”的系统。
